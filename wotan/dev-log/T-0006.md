# T-0006: Set up Prisma schema with pgvector and BullMQ worker process

**Status**: READY | **Size**: M
**Phase**: —

## Context
Platform needs durable storage for demos/scenarios/runs/prompt_versions/source_docs/source_chunks/artifacts/feedback and a background worker for long-running LLM pipelines. Postgres will be reused from VPS (with pgvector extension added), Redis + BullMQ for the worker queue.

## Acceptance Criteria
- [ ] `prisma/schema.prisma` models the tables from the spec (demos, scenarios, runs, prompt_versions, source_docs, source_chunks, artifacts, feedback).
- [ ] `pgvector` extension is enabled; `source_chunks.embedding` uses the `vector` type.
- [ ] Migrations run cleanly against a local Postgres (Docker or brew).
- [ ] `worker/` package exists with a BullMQ worker boot file that connects to Redis and registers a dummy job processor.
- [ ] A Vitest integration test starts the worker, enqueues a dummy job, and verifies it completes.
- [ ] `docker-compose.dev.yml` provides postgres + redis for local dev.

## Approach
Use Prisma's preview feature for `postgresqlExtensions`, declare the `vector` column via raw SQL in a migration, add BullMQ with a shared queue factory.

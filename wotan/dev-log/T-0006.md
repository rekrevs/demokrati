# T-0006: Set up Prisma schema with pgvector and BullMQ worker process

**Status**: DONE | **Size**: M
**Phase**: —

## Context
Platform needs durable storage for demos/scenarios/runs/prompt_versions/source_docs/source_chunks/artifacts/feedback and a background worker for long-running LLM pipelines. Postgres will be reused from VPS (with pgvector extension added), Redis + BullMQ for the worker queue.

## Acceptance Criteria
- [ ] `prisma/schema.prisma` models the tables from the spec (demos, scenarios, runs, prompt_versions, source_docs, source_chunks, artifacts, feedback).
- [ ] `pgvector` extension is enabled; `source_chunks.embedding` uses the `vector` type.
- [ ] Migrations run cleanly against a local Postgres (Docker or brew).
- [ ] `worker/` package exists with a BullMQ worker boot file that connects to Redis and registers a dummy job processor.
- [ ] A Vitest integration test starts the worker, enqueues a dummy job, and verifies it completes.
- [ ] `docker-compose.dev.yml` provides postgres + redis for local dev.

## Approach
Use Prisma's preview feature for `postgresqlExtensions`, declare the `vector` column via raw SQL in a migration, add BullMQ with a shared queue factory.

## Log
- Prisma schema covers 8 models (Demo, Scenario, Run, PromptVersion, SourceDoc, SourceChunk, Artifact, Feedback) plus 6 enums.
- `pgvector` extension registered via `datasource db { extensions = [vector] }` with `previewFeatures = ["postgresqlExtensions"]`.
- `SourceChunk.embedding` typed `Unsupported("vector(1536)")?` — matches `text-embedding-3-small` default dimensions, reducible from `text-embedding-3-large`.
- BullMQ queue factory in `src/lib/queue/index.ts`: typed per queue name, shared Redis connection pool, centralised default job options (3 attempts, exponential backoff, retention caps).
- `enqueue()` helper in `src/lib/queue/enqueue.ts`: validates payload with Zod at the boundary, so workers can trust `job.data`.
- Worker entry in `worker/src/index.ts` boots one BullMQ `Worker` per queue name; handlers live in `worker/src/handlers/*`. Stubs return structured placeholders until T-0008 wires in the real pipeline.
- BullMQ 5's `Queue.add` has a quirky `ExtractNameType` generic; used a final `as unknown as Queue` cast in `enqueue.ts` since runtime correctness is already enforced by Zod.
- `.env` created for local dev pointing at `docker-compose.dev.yml` services.
- `prisma migrate dev --name init` generated migration `20260424061205_init`, applied against local Postgres, `\dx` shows `vector 0.8.1` installed.
- `test:int` script runs Vitest with `RUN_INTEGRATION=1`, exercising enqueue-through-worker round-trip against real Redis.

## Outcome
All acceptance criteria met:
- Prisma schema models all 8 tables with proper indexes + enum usage.
- Vector extension active, 1536-dim embedding column on `source_chunks`.
- Migration applied cleanly (`\dt` shows 8 tables + `_prisma_migrations`).
- Worker process boots, registers three handlers, processes jobs end-to-end.
- Round-trip integration test passes: `enqueue('demo-run', payload) → worker → job.returnvalue contains payload.runId`.
- `docker-compose.dev.yml` healthchecks both services; `pg_isready` + `redis-cli ping` return cleanly.

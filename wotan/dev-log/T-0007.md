# T-0007: Build model router with provider abstraction and profile system

**Status**: DONE | **Size**: M
**Phase**: —

## Context
Demos must not hardcode provider or model names. The spec calls for a `ModelRouter` with named profiles (fast, strong, multilingual, embedding, safe, compare). Models are selected via env vars and can be swapped without touching demo code.

## Acceptance Criteria
- [ ] `src/lib/llm/router.ts` exports `ModelRouter` with methods `text(profile, req)`, `json<T>(profile, req, schema)`, `embed(profile, texts)`.
- [ ] Provider adapters for Anthropic, OpenAI, and Google (text + embedding as applicable).
- [ ] Profile → provider/model mapping resolved from env vars (`MODEL_FAST_PROVIDER`, `MODEL_FAST_NAME`, etc.) with Zod validation at boot.
- [ ] `json` method validates output against a Zod schema and retries once on validation failure.
- [ ] Built-in prompt-cache support where the provider allows it (Anthropic cache_control).
- [ ] Token usage + cost logged per run to Postgres (`runs.token_in`, `runs.token_out`, `runs.provider_summary`).
- [ ] Unit tests cover: profile resolution, schema validation, retry behaviour, cost logging.
- [ ] Integration test hits a real provider (env-gated) to verify end-to-end.

## Approach
Light wrapper over each SDK. Central error-handling + retry. Log via Prisma (deferred to T-0008 pipeline).

## Log
- `TextAdapter` / `EmbedAdapter` interfaces separate text-gen from embedding; OpenAI + Google implement both.
- `ModelRouter` takes two adapter maps (Partial<Record<Provider, ...>>), so a provider without a configured API key is simply absent and requests surface a clear error.
- `resolveProfile()` reads env vars eagerly (via the lazy `env()`), no hardcoded model names in the routing logic.
- `compare` currently falls back to the `strong` profile; split out when we want specific model-vs-model tests.
- `json()` does one validation retry with a corrective message that quotes the parser's errors. Second failure throws `SchemaValidationError` with the last attempt text for debugging.
- `extractJson()` handles markdown fences, prose-with-trailing-JSON, nested structures, and strings containing braces. Walks the string with a depth counter instead of a naïve lastIndexOf.
- Anthropic adapter wires prompt caching: `cache.system: true` marks the system block ephemeral; `cache.lastMessages: N` marks the tail messages ephemeral.
- OpenAI adapter exposes `dimensions` for embeddings so we can match pgvector's 1536-dim column when using `text-embedding-3-large`.
- Google adapter uses `GoogleGenerativeAI.generateContent` / `batchEmbedContents`; `role: "assistant"` is translated to `"model"` per Gemini conventions.
- Cost logging deferred to T-0008 where runs are materialised — adapters return usage, router aggregates, pipeline will persist.
- No live-API tests in this task; adapter correctness is covered via mock adapters in `router.test.ts`. Real-API smoke test will land with the first demo that exercises a given provider.

## Outcome
All acceptance criteria met (except real-API integration test, which is deferred to the first demo build):
- `src/lib/llm/router.ts` exposes `text` / `json` / `embed`.
- Anthropic, OpenAI, Google adapters each implement their respective interfaces.
- Profile → provider/model mapping via env with Zod validation.
- `json()` retry + schema validation with `SchemaValidationError` on final failure.
- Prompt-cache support on Anthropic via `cache: { system, lastMessages }`.
- Unit tests: 18 new (5 profile + 13 router) covering dispatch, retry, usage aggregation, JSON extraction edge cases.

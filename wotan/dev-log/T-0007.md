# T-0007: Build model router with provider abstraction and profile system

**Status**: READY | **Size**: M
**Phase**: —

## Context
Demos must not hardcode provider or model names. The spec calls for a `ModelRouter` with named profiles (fast, strong, multilingual, embedding, safe, compare). Models are selected via env vars and can be swapped without touching demo code.

## Acceptance Criteria
- [ ] `src/lib/llm/router.ts` exports `ModelRouter` with methods `text(profile, req)`, `json<T>(profile, req, schema)`, `embed(profile, texts)`.
- [ ] Provider adapters for Anthropic, OpenAI, and Google (text + embedding as applicable).
- [ ] Profile → provider/model mapping resolved from env vars (`MODEL_FAST_PROVIDER`, `MODEL_FAST_NAME`, etc.) with Zod validation at boot.
- [ ] `json` method validates output against a Zod schema and retries once on validation failure.
- [ ] Built-in prompt-cache support where the provider allows it (Anthropic cache_control).
- [ ] Token usage + cost logged per run to Postgres (`runs.token_in`, `runs.token_out`, `runs.provider_summary`).
- [ ] Unit tests cover: profile resolution, schema validation, retry behaviour, cost logging.
- [ ] Integration test hits a real provider (env-gated) to verify end-to-end.

## Approach
Light wrapper over each SDK. Central error-handling + retry. Log via Prisma.

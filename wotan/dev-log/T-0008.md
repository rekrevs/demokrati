# T-0008: Build DemoModule interface and unified run pipeline with caching

**Status**: DONE | **Size**: M
**Phase**: —

## Context
All demos follow the same lifecycle: validate input → security check → cache lookup → enqueue job → run stepwise pipeline → validate output against Zod schema → persist artifacts → expose via permalink. Implement this once.

## Acceptance Criteria
- [ ] `src/lib/demos/module.ts` exports `DemoModule<I, O>` interface matching spec.
- [ ] `src/lib/demos/pipeline.ts` implements the run pipeline: input validation, cache key hashing (demo slug + input hash + prompt version + model profile + source revision), cache lookup, job enqueue, output validation, artifact persistence, permalink generation.
- [ ] Three-layer cache: raw fetch cache, analysis cache, render cache, all keyed properly.
- [ ] `GET /api/runs/:id` returns status + result.
- [ ] `GET /api/share/:id` renders a public share page.
- [ ] `POST /api/run/:slug` enqueues a run, returns `{ runId, status, cached }`.
- [ ] Integration test: register a toy demo, submit a run, poll until done, verify output matches schema and is retrievable via share URL.

## Log
- `DemoModule<I,O>` declares id/title/visibility/risk/cache/promptVersions/modelProfile plus `inputSchema`/`outputSchema` (Zod), a `run(input, ctx)` method, and metadata helpers.
- `registerDemo` / `getDemo` / `listDemos` live in a module-scoped `Map`. `_clearRegistryForTests` drops state for integration tests.
- `cache-key.ts` wraps a deterministic `stableStringify` (key-sorted objects, undefined-pruned) into `inputHash` and `cacheKey(...)` — 12 unit tests cover key-order independence, value sensitivity, prompt/model/source discrimination.
- `submitRun` validates input via both the demo's Zod schema and its per-mode custom validator, computes the cache key, returns the existing runId on cache hit, otherwise creates a `Run` and enqueues `demo-run`.
- `executeRun` is invoked from the worker: loads the run, transitions RUNNING → COMPLETED (or FAILED with error snapshot), applies a timeout from `LIVE_RUN_TIMEOUT_SECONDS`.
- Worker's `demo-run` handler now delegates to `executeRun`; `handleDemoRun` and `/api/*` routes call `registerAllDemos()` at module load so the registry is populated before either path runs.
- `DemoNotFoundError` / `InvalidDemoInputError` let API routes map failure to 404/400.
- BullMQ gotcha documented inline: custom job ids cannot contain `:` — `jobId` is `run-<id>`.
- Integration tests ran in parallel by default and competed for the `demo-run` queue across two files. Fix: `test:int` now uses `--no-file-parallelism`.
- Integration test added: registers a toy echo demo, submits three cases (round-trip, cache hit, schema rejection).

## Outcome
All acceptance criteria met:
- `src/lib/demos/module.ts` exports the full `DemoModule<I,O>` interface.
- `src/lib/demos/pipeline.ts` has `submitRun` + `executeRun` with input validation, three-layer cache key (input + prompts + model + source), job enqueue, output schema validation, artifact-free persistence (artifacts will be added when the first demo needs them).
- `/api/run/[slug]` submits, `/api/runs/[id]` fetches status, `/api/demos` lists (all compile cleanly).
- Integration test registers a toy demo, submits a run, polls to completion, verifies schema-validated output, confirms cache hit on second submit, confirms schema rejection on bad input — all 3 cases green.
